# ADR-002: LLM Agent Code Generation for Product Family Onboarding

## Status

Accepted

## Context

The constraint engine needs to scale from 3 product families to 13+. Each family requires Python artifacts: Pydantic models, constraint rules, config, loader, sample data, and tests. Two approaches were evaluated:

**Option A: Schema-driven declarative rules.** Products defined as JSON Schema, rules as JSON definitions (equality, range, membership, etc.), evaluated by a generic rule engine. No Python per family. See [DESIGN-generic-schema-driven-solver.md](../../design/DESIGN-generic-schema-driven-solver.md) for the full exploration.

**Option B: LLM agent code generation.** Domain expert describes the family in plain English. A specialised agent generates Pydantic models, rule functions, config, loader, and tests as standard Python. Validated through an automated pipeline before human review.

Analysis of the 31 existing rules showed ~90% fit 6 declarative types — but the remaining 10% (overlay lookups, compatibility tables, computed expressions) require either an escape hatch to Python or an increasingly complex expression language. Meanwhile, all three current families were successfully generated by an LLM in this project with no hand-coding.

## Decision

Use LLM agent code generation for product family onboarding. Keep constraint rules as Python code. Do not build a schema-driven declarative rule engine.

The approach:

1. **A specialised family-builder agent** generates all Python artifacts from a domain expert's natural language description of the product family.

2. **A 6-step validation pipeline** ensures correctness:
   - Requirements gathering (structured interview)
   - Model generation with field/type/enum validation
   - Rule generation with cross-referencing against models
   - Sample data and golden test generation
   - Cross-validation (all tests pass, every rule exercised, no unreachable rules)
   - Human review of constraint traces for golden scenarios

3. **Generated code is standard Python** — Pydantic models, pytest tests, git version control. No custom tooling, no declarative rule evaluator, no expression language.

4. **Runtime rule parameterisation** (thresholds, lookup tables as data, rule logic as code) remains an option for the narrow case where operational staff need to adjust values without a deployment. This is a smaller, more focused version of "rules as data" that avoids the expression language problem.

## Consequences

### What becomes easier

- **Adding families requires no framework development.** No declarative rule evaluator to build, maintain, or extend. The existing `NCandidateSolver` and N-candidate rule signature are sufficient.

- **Complex rules are just code.** Overlay lookups, compatibility tables, computed expressions — all expressible directly in Python without escape hatches or expression language extensions.

- **Debugging is standard.** Breakpoints, stack traces, pytest — developers use tools they already know. No custom debugging for declarative rule definitions.

- **Type safety preserved.** Pydantic catches field errors at import time. No risk of JSON Schema path typos failing silently at runtime.

- **Validation is more robust than hand-coding.** The agent pipeline systematically cross-validates models against rules, generates tests that exercise every rule in both pass and fail states, and verifies solver integration. A developer might skip edge cases; the agent always checks.

- **The pattern is proven.** Three families were generated in this project: drawer slides (N=1), LED lighting (N=3), and concealed hinge v2 port (N=2, 14 rules). All with tests passing on first generation.

### What becomes harder

- **No runtime rule editing.** Changing a rule requires code generation, testing, and deployment. This is intentional — for a system where correctness is the core promise, a deploy gate with tests is a feature. But it means a sales manager can't adjust a weight threshold in a dashboard.

- **Depends on LLM capability.** The approach assumes LLMs can reliably generate correct Pydantic models and constraint rules from natural language. This is true today for Claude and GPT-4 class models. If model quality regresses, the approach weakens.

- **Agent pipeline needs building.** The specialised family-builder agent with its 6-step validation pipeline is a development effort — though a smaller one than building a declarative rule evaluator with expression language, custom testing, and debugging tools.

- **Domain expert must review.** The human review step is essential — the agent generates code, but the domain expert must confirm the constraint traces match reality. This is the same requirement as declarative rules; neither approach eliminates the need for SME validation.
